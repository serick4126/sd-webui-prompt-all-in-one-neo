# プロンプト表示エリアのリアルタイム更新 設計仕様

- 日付: 2026-06-16
- 対象拡張: sd-webui-prompt-all-in-one-neo
- 主対象ファイル: `src/src/components/phystonPrompt.vue`, `src/src/utils/splitTags.js`

## 1. 背景と問題

プロンプト入力エリア(Gradioの `<textarea>`)を編集すると、その内容がプロンプト表示エリアにタグとして反映される。自動更新(`autoLoadWebuiPrompt`)を有効にしていても、入力中はリアルタイムに反映されず、表示エリアのクリック等でフォーカスが外れて初めて更新される。これが使いづらく、リアルタイムに近い更新を実現したい。

ただし、完全な毎キー更新は別の問題を生む。末尾チップのラベルが `w→wa→…→walking` と伸び、横幅制限を超えて折り返すと、**入力中に表示エリアのサイズが変わり、後続タグの位置が1文字ごとにずれ続ける**(特に長いプロンプトの途中にタグを挿入する場合)。これは非常に見づらい。

## 2. 非リアルタイムの真因

現状は input イベントではなく **500ms ポーリング**で textarea を監視し、さらに次の条件で入力中は明示的に更新をスキップしている(`phystonPrompt.vue` 959-968 付近)。

```js
if (document.activeElement === this.textarea) return  // 入力中はスキップ
```

「クリックで更新される」のは、クリックでフォーカスが外れポーリングが拾うため。スキップが入っているのは、更新処理(`updateTags` → `updatePrompt`)が**タグ再描画だけでなく textarea へ文字列を書き戻す**ため。入力中に書き戻すとキャレット位置がリセットされ入力と干渉する。フォーカス中スキップはこの回避策と考えられる。

したがって問題は2層に分かれる。

1. **何を更新するか**(表示だけか、textarea も書き戻すか)= キャレット問題の有無を決める層
2. **いつ発火するか**(構造の区切り検知)= リフローを起こさないための層

## 3. 設計目標 / 非目標

### 目標

- 入力中にタグ表示が「ほぼリアルタイム」に追従する。
- 入力中に後続タグが1文字ごとにずれるリフローを起こさない。
- ペースト・カンマ削除による結合・オートコンプリート補完・IME入力でも正しく動く。

### 非目標(YAGNI)

- 入力中の textarea への書き戻し(キャレット維持処理ごと不要と判断)。
- リアルタイム更新の ON/OFF トグル UI(設計上は容易だが今回は実装しない)。
- 表示エリア側からの編集と入力中更新の同時並行(入力中に表示エリアを操作するユーザーは想定しない)。

## 4. 中核アーキテクチャ(2経路の併存)

```
[経路1: 既存] 500ms ポーリング … フォーカス外 / 外部書き換え用
   - document.activeElement !== textarea のときだけ動く(現状維持)
   - blur 時は従来どおり「全タグ化 + textarea 書き戻し」のフル更新

[経路2: 新規] input リスナー … フォーカスして入力中の経路
   - 一方向(textarea → タグ表示)のみ。書き戻しは一切しない
   - 「確定済みセグメント」だけを描画し、編集中セグメントは凍結
```

経路2は `document.activeElement === textarea` の間だけ働き、既存ポーリングのスキップ条件と住み分ける。**blur した瞬間に経路1のフル更新へ引き継ぎ**、そこで初めて末尾の編集中トークンもタグ化され、textarea 書き戻しも従来どおり実行される。

## 5. 確定検知アルゴリズム(案B: キャレット基準・構造監視)

`input` 発火ごとに以下を行う(すべて一方向・表示のみ)。

1. **範囲付き分割** — `splitTags` を拡張(または併走関数)し、各セグメントを文字列に加えて `[start, end]` オフセット付きで返す。括弧 `()` `[]` `<>` `{}` のバランス追跡は既存ロジックを流用。**バランスが取れていないセグメントは未確定**扱い。
2. **編集中セグメントの特定** — `textarea.selectionStart`(キャレット位置)がどのセグメント範囲に入るかを引く。
3. **確定/編集中の判定** — 後述の「確定の定義」に従い、編集中セグメントとそれ以外を分ける。
4. **タグ配列の構築** — 確定済みセグメントは現在の内容で描画。編集中セグメントは「凍結スナップショット」で描画(後述)。各タグに**安定キー**を付与。
5. **Vue に委ねる** — 描画対象集合が前回と同一なら Vue の差分はゼロ → DOM を一切触らない → リフローなし。確定済み集合が変わった時だけ再描画。

この方式は「何を打ったか」を一切見ず「**結果としての構造**」だけを見るため、ペースト・補完・IME経由カンマ・カンマ削除結合がすべて自動で正しく扱われる。

### 確定の定義(本フローの中核ルール)

> あるセグメントが新しい内容でタグ化される=「確定」する瞬間とは、そのセグメントが「編集中」でなくなった時。
>
> **「編集中」= キャレットがそのセグメント内にあり、かつ内容が前回タグ化時と異なる(または括弧が未バランス)。**

確定の契機は次のいずれか。

- **キャレットがそのセグメントの外へ出る**(中身編集の主役の契機)
- **未バランスの括弧が閉じてバランスする**(`{...` → `}`)
- **blur**(フォーカス喪失 → 経路1のフル更新へ引き継ぎ)

末尾追記の「カンマで確定」も、この定義の一種である。カンマが新セグメントを作り、キャレットが直前セグメントの外(後続の新セグメント)へ移動する結果として確定する。カンマ自体が確定トリガーなのではない。末尾追記も中身編集も「**キャレットがセグメントから出たら確定**」という1ルールに統一される。

クリックでセグメントに入っただけ(内容が前回タグ化時と同一)では編集中にならない。最初に1文字変更した時点で初めて編集中になる。

## 6. レンダリングモデル(凍結スナップショットと安定キー)

編集中セグメントの見せ方は「**旧内容で凍結表示**」を採用する。

- セグメントが「確定 → 編集中」に遷移した時点で、その時の確定内容を**スナップショット**として捕捉する。
- 編集中の間、そのチップは**スナップショットの内容を固定幅で描画**し続ける(実テキストとは一時的に不一致)。チップの幅が変わらないためリフローが起きない。
- 確定(キャレットが外へ出る等)した時点でスナップショットを解放し、新しい内容で描画する。

派生する重要な性質。

- **新規末尾トークンはスナップショットを持たない**(確定済みの前内容が存在しない)ため、チップは出ない。これは「確定までタグ化しない」要件と一致する。
- **キャレットは1つ**なので、凍結チップは常に最大1個。状態は「現在の編集中セグメントのキーと凍結内容」1組で表現できる。
- セグメント内にカンマが追加/削除されて**構造自体が変わった場合**はスナップショットを解放し、新しい構造で通常どおり再描画する。

### 安定キー

確定済みタグには、再パースをまたいで継続する**安定キー**を付与する(セグメントのオフセット連番のような揺れる値ではなく、確定テキストに基づく値、もしくは既存タグオブジェクトの再利用によるキー継続)。これにより、編集中セグメント内をいくら打っても確定済みタグのキーが振り直されず、Vue が DOM を触らない。**安定キー設計と「確定済み集合が不変なら DOM を触らない」の2点がリフロー封じの肝**である。

## 7. エッジケース

- **IME(日本語変換)** — `compositionstart` 〜 `compositionend` の間は再描画を抑止し、`compositionend` で1回フラッシュ。変換中は `selectionStart` も未確定文字列も信用できないため「触らない」。
- **ペースト / カンマ削除による結合・分割** — 案Bが結果の構造だけを見るため追加対応不要。確定済み集合が変われば1回だけ再描画。
- **末尾の空セグメント**(`cat, ` の直後)— キャレットが乗る編集中セグメントとして扱われ、スナップショットが無いのでチップは出ない。
- **blur 引き継ぎ** — フォーカスが外れたら経路1のフル更新へ。末尾の編集中トークンもタグ化され、textarea 書き戻しも実行。
- **性能** — 長いプロンプトで毎 input 時にパースが走る。フル再構築の前に「編集中セグメントをマスクした文字列シグネチャ」を前回と比較し、不変なら即 return する安価な前段ガードを入れ、パース・差分コストを抑える。

## 9. 設定 / トグル

既定 ON とする。旧挙動を残したい場合に備えてトグル化は容易(`autoRealtimeUpdate` のような prop を追加するだけ)だが、今回は実装しない。設計上の逃げ道として記録するに留める。

## 10. テスト方針

- **リフロー非発生**: 長いプロンプトの先頭付近のタグを編集しても、後続タグの位置が1文字ごとに動かないこと(確定済み集合が不変な間は DOM 変化ゼロ)。
- **確定タイミング**: 末尾追記でカンマを打つと直前トークンがタグ化される / 中間タグ編集はキャレットが外へ出た時にタグ化される。
- **凍結表示**: `outdoor, dog, mountain` の `dog` を `dig` に編集中、チップは `dog` のまま固定幅で残り、キャレットが外へ出た瞬間 `dig` に更新される。新規末尾トークンはチップが出ない。
- **構造変化の堅牢性**: ペースト、カンマ削除による結合、IME 経由のカンマ入力で確定済み集合が正しく更新される。
- **IME**: 変換中は再描画されず、確定(`compositionend`)で1回だけ反映される。
- **既存経路の非回帰**: フォーカス外の外部書き換えは従来どおりポーリングで反映、blur 時のフル更新と書き戻しが従来どおり動く。
